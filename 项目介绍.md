# Aura AR 项目介绍

## 1. 项目概览

这是一个围绕 AR 头显展开的多模态交互项目，核心目标不是单点功能演示，而是把光学显示、底层渲染、空间交互、导航、语音、声音感知和后视辅助整合到同一套低延迟系统中。项目主控程序为 `ar_app1_hud.cpp`，运行在 ELF2 / RK3588 侧，配套包含手语识别、导航通信、语音识别、离线语音合成、声音雷达和后置摄像头等模块。

从系统结构看，这个项目有几个非常鲜明的特点：

- 显示链路不是跑在传统桌面窗口之上，而是直接深入到底层 DRM / GBM / EGL / OpenGL ES 渲染栈，以减少不必要的桌面合成开销。
- 光学链路不是“把画面画出来就结束”，而是必须先对近眼光学畸变做反向校正，再解决双目汇聚与双目视差问题，最后才能在真实佩戴状态下得到自然画面。
- 输入链路不是单一交互，而是同时接入了 Ultraleap 双目红外手部跟踪、手机导航、8 路麦克风声音感知、后置摄像头和语音识别/语音播报。
- 模型部署强调本地化、低时延与离线能力，STT/TTS 接入 sherpa-onnx，手语模型和声音分类模型接入 RKNN / RKNNLite，充分利用 RK3588 平台的边缘推理能力。

## 2. 系统总体架构

项目可以理解为一个“统一 AR 运行时”：

1. `ar_app1_hud.cpp` 负责主显示、状态机、应用切换、底层渲染和多线程调度。
2. 光学部分通过 `calibration.json` 中的标定参数构建双目畸变网格，将逻辑 UI 映射到真实可视平面。
3. 手语模块由 `ultraleap_rknn_live` 负责采集 Sir170 的 3D 手部骨架并调用 `sign_rk3588.rknn` 做推理。
4. 导航模块由 Flutter 手机 App `aura_nav` 提供路径规划、定位与蓝牙通信，头显侧通过 Bluetooth RFCOMM 接收导航 JSON 数据并绘制 HUD 界面。
5. 语音模块分两部分：`sherpa-onnx-alsa` 负责语音转文字，`sign_translate_tts_service.py` + `sherpa-onnx-offline-tts` 负责离线文字转语音。
6. 声音雷达由 `sound_radar_service.py` 完成采集、分类、定位，并把结果通过本地 TCP 回传给 HUD。
7. 后置摄像头模块通过 V4L2 / OpenCV 取流，再由 HUD 统一合成到骑行模式界面中。

这种架构的优点是：各功能都以独立服务/线程运行，但最终统一汇总到一个近眼渲染和交互系统里，既保证了模块边界清晰，也保证了最终用户体验是一体化的。

## 3. 光学系统与底层显示链路

### 3.1 光学方案：基于 North Star 思路的双目近眼显示

本项目采用的是 North Star 风格的双目近眼光学方案。公开资料中，Project North Star 是一套强调大视场、开放硬件和手部交互结合的 AR 头显设计，核心思想是：

- 使用双目显示系统分别向左右眼送入图像；
- 通过近眼光学结构把显示内容投送到人眼可接受的观察空间；
- 在较大视场下追求更自然的空间叠加感；
- 与手部追踪系统天然结合，强调“看见 + 看懂 + 可交互”。

但这类方案真正落地时，最难的从来不是“把屏幕点亮”，而是“让人眼看到正确画面”。你的项目在这方面实际上已经做到了比较完整的一条链路。

### 3.2 难点一：镜片本身存在畸变，必须先做反畸变校正

近眼光学系统的第一大难点，是镜片/组合光路自身就会引入明显几何畸变。直接把 2D UI 生硬绘制到面板上，经过镜片后，人眼看到的图像会变形，文字和图标会扭曲，双目情况下还会进一步放大不适感。

在 `ar_app1_hud.cpp` 中，这一问题是通过标定数据驱动的“每眼 UV 反畸变网格”来解决的。关键实现包括：

- 读取 `calibration.json`；
- 解析 `left_uv_to_rect_x / left_uv_to_rect_y / right_uv_to_rect_x / right_uv_to_rect_y` 等多项式系数；
- 根据每只眼的标定参数生成高密度 distortion mesh；
- 在渲染阶段通过网格顶点的 `u,v` 重映射，把逻辑画面扭成“经过镜片后看起来才正常”的图像。

从代码逻辑看，这套做法不是简单的平面仿射变换，而是基于标定多项式对每个采样点的光线进行重建，再计算其落到显示平面上的采样坐标。也就是说，你实际做的是“先从人眼观察空间反推到显示空间”的校正思路。

### 3.3 难点二：双目汇聚、双目视差与校正数据耦合

第二个难点比反畸变更棘手：左右眼不仅各自有畸变，还要在矫正之后实现稳定的双目汇聚。

你的代码里并不是把左右眼简单复制一份 UI，而是进一步引入了：

- 左右眼不同的光学平面角度；
- 左右眼各自的 `eye_origin_x_mm`；
- 倾斜显示平面的几何求交；
- 面板剪切量 `panel_shear`；
- 可调的 `g_symmetric_shear` 运行时参数。

这意味着双目汇聚在你的系统中不是“后处理贴个偏移”这么简单，而是把以下几件事耦合在一起考虑：

1. 标定得到的左右眼 UV 校正数据；
2. 观察光线与倾斜显示面的真实相交关系；
3. 左右眼对称但方向相反的 plane angle / panel shear；
4. 为了佩戴体验而保留的运行时微调能力。

换句话说，你的双目汇聚并不是单纯的几何偏移，而是把“畸变校正”和“会聚调参”统一到一个渲染数学模型里。这正是近眼 AR 系统最关键也最难做好的一步。

### 3.4 UV 平面校正与可调收敛

代码中 `getEyeAdjustments()`、`intersectTiltedPlane()`、`computePanelUv()` 和 `generateDistortionMeshesHudFromCalibration()` 这几部分串起来后，形成了一条完整的链路：

- 先由标定多项式恢复每只眼的视线方向；
- 再与倾斜显示平面求交；
- 再把交点映射回面板局部坐标；
- 最后得到真正用于采样的 `src_u/src_v`。

同时，`g_symmetric_shear` 可以在运行时修改，一旦参数变化，双目 distortion mesh 会重新生成。这让系统具备了“调参即更新光学校正”的能力，非常适合真实头显调机阶段使用。

### 3.5 驱动层细节：绕开桌面系统，直接操作底层硬件

为了追求更高性能，这个项目关闭桌面系统，直接从底层图形硬件进行操作。主程序中可见：

- 直接访问 `/dev/dri/card0`；
- 使用 DRM / GBM 完成缓冲区管理；
- 使用 EGL / GLES2 进行渲染；
- 使用 NanoVG 完成高质量矢量 UI 绘制。

这么做的价值很明确：

- 避免桌面合成器带来的额外延迟；
- 降低图像链路抖动；
- 更容易把光学校正网格、双目渲染和 HUD UI 合在一起管理；
- 更适合面向 AR 头显的固定功能运行时。

你在 UI 层选择 NanoVG 也很合理。对这种需要大量线条、圆角、描边、渐变、平滑动画和可读文字的 AR HUD 来说，NanoVG 在抗锯齿和精细轮廓表现上明显优于“把 OpenCV 当 UI 库来画”的方案。

## 4. 手语识别功能

### 4.1 硬件基础：Ultraleap Sir170 双目红外相机

手语识别部分使用的是 Ultraleap Sir170 双目红外相机。它的优势不是普通 RGB 摄像头能替代的：

- 天然面向近距离手部交互；
- 可以实时输出高精度 3D 手部骨架；
- 在毫米级空间尺度上提供稳定的手部关节点坐标；
- 对近眼交互特别友好，因为它不是只做 2D 检测，而是直接提供 3D 重建结果。

在你的系统里，Sir170 不只是“看见手”，而是为后续手语识别模型直接提供骨架级别的结构化输入。

### 4.2 手语识别后端与模型

手语识别实际由 `ultraleap_rknn_live` 执行，它把 LeapC 采集到的骨架数据预处理后送入 RK3588 NPU 运行 `sign_rk3588.rknn`。

从代码可以读出，这个模型的输入结构是：

- 2 只手；
- 26 个关节点；
- 每点 3 个空间坐标；
- 额外拼接位置与速度，共 6 个输入通道；
- 时间维统一重采样到 60 帧；
- 最终张量为 `[1, 6, 60, 26, 2]`。

这说明你的模型不是做“单帧静态姿态分类”，而是典型的时序手语识别。它既看空间结构，也看动作随时间的演化。

### 4.3 识别流程

`ultraleap_rknn_live` 的流程很有代表性：

1. 连接 Ultraleap 设备；
2. 使用 `eLeapTrackingMode_HMD` 进入头显跟踪模式；
3. 持续接收手部帧；
4. 直到第一只右手出现后，开始建立一个手势窗口；
5. 以第一帧右手掌心为锚点，对后续节点做相对化处理；
6. 当 0.3 秒内没有新的手帧时，关闭当前手势窗口；
7. 把窗口序列整理成 `[1,6,60,26,2]` 输入；
8. 交给 RKNN 模型完成分类。

这种“以右手起始掌心为锚点”的做法很重要，因为它把用户在不同空间位置做出的手势变成了更稳定的相对动作模式，降低了绝对位姿变化对分类结果的影响。

### 4.4 模型类别

从 `labels.c` 可以读出当前模型是一个 15 类手语模型，包含如下拼音标签：

- `shangxin`
- `nihao`
- `qu`
- `chi`
- `mingzi`
- `dajiahao`
- `haipa`
- `gongzuo`
- `kaixin`
- `mang`
- `gandong`
- `unknown`
- `lai`
- `xianzai`
- `kan`

也就是说，这个模型已经不只是“左右挥手/简单手势”的演示，而是具备基础词汇表达能力，可以进一步与听障交流和语音播报模块联动。

### 4.5 与 TTS 的耦合

在你的 AR 系统里，手语识别不是一个孤立模块。识别结果会进一步送入 TTS 线程，使用本地离线语音合成播报出来。这样一来，Sir170 + 手语模型 + TTS 共同构成了完整的“手语输入 -> 语音输出”链路，直接服务听障交流场景。

## 5. 导航功能

### 5.1 头显侧导航设计

HUD 端导航不是传统 2D 地图平移，而是专门为近眼显示设计的简洁导航界面。代码中可以看到它包含多个互相协同的视觉层：

- 前方主引导路径；
- 蓝色流光式路线高亮；
- 车辆/朝向 puck；
- 小地图总览；
- 转向箭头；
- 剩余距离进度条；
- 到达态和待机态切换动画。

这套 UI 的重点不是堆信息，而是减少认知负担。近眼显示的面积和注意力窗口都很有限，因此真正有用的是：

- 把“下一步动作”放到最醒目位置；
- 把“全局路线”做成辅助；
- 把“剩余距离”和“转弯距离”做成低认知负担数字信息；
- 用平滑过渡和方向滤波避免箭头乱跳、路径抖动和方向突变。

代码里对导航平滑做了多层处理：

- 路径只在 `full_path` 更新时重建；
- 小车朝向使用 `smooth_heading` 逐步收敛到 `raw_heading`；
- 当前显示距离与转向距离使用 display 变量做过渡；
- UI 状态从 `STANDBY -> TRANSITION_IN -> NAVIGATING -> ARRIVED` 分层切换。

这让界面不是“硬切换的信息面板”，而更像一个持续、平稳、可读的 AR 导航仪表。

### 5.2 手机 App：导航系统真正的智能核心

虽然头显侧负责最终显示，但从工程量和逻辑复杂度来看，手机端 `D:\Flutter_projects\aura_nav` 才是导航功能的智能核心，值得重点介绍。

这个 App 主要负责四件事：

1. 定位与定位稳态管理；
2. 高德路线规划；
3. 地图搜索、选点和路线缓存；
4. 通过蓝牙把导航状态持续同步给头显。

### 5.3 Flutter App 的定位与路径规划逻辑

App 使用了：

- `amap_flutter_map` / `amap_flutter_base`：地图显示和基础地图能力；
- `Geolocator`：高精度位置获取；
- `permission_handler`：Android 权限处理；
- 自定义 `BluetoothTransport`：蓝牙 MethodChannel 通道。

定位方面，它不是只读一次位置，而是做了“持续位置流 + 定期强制刷新”的双通路设计：

- `LocationAccuracy.bestForNavigation`；
- `distanceFilter = 2` 米；
- Android 侧强制导航级定位；
- 周期性刷新当前位置；
- 当有新位置且正在导航时，触发路径更新逻辑。

路径规划方面，它会请求高德骑行路径接口：

- 根据起点和目标点获取骑行路径；
- 对每个导航 step 解析 `polyline`；
- 合并成完整路径；
- 计算每个 step 的起止累计距离；
- 建立 `_CachedRoute` 供后续快速投影和重路由使用。

### 5.4 路径投影与重路由机制

这部分是手机端最值得强调的地方之一。

App 并不是每次位置更新都粗暴重新请求路线，而是先做几何投影：

- 把当前位置投影到当前路线折线段上；
- 计算当前已经走过的进度 `progressMeters`；
- 计算当前位置到路线的最短距离 `distanceToRouteMeters`；
- 如果偏离路线大于 45 米且连续两次命中，则触发重路由。

这套机制有两个优点：

1. 减少无意义的网络请求和路线抖动；
2. 让 HUD 收到的“下一步动作 / 剩余距离 / 总距离”更稳定。

### 5.5 蓝牙交互：手机与头显的桥梁

手机端蓝牙层由 `bluetooth_transport.dart` 封装，通过 MethodChannel `aura_nav/bluetooth` 调用 Android 原生蓝牙能力。App 支持：

- 判断设备是否支持蓝牙；
- 判断蓝牙是否开启；
- 获取已配对设备列表；
- 用户选择 ELF2 板端蓝牙地址；
- 连接 / 断开；
- 向头显发送 JSON 负载。

头显端则在 `ar_app1_hud.cpp` 中开启 Bluetooth RFCOMM 服务，注册 SDP 服务描述，并在 socket 中按行接收 JSON。这样，手机端就能把如下信息同步给头显：

- 当前经纬度；
- 目的地经纬度；
- 目的地名称；
- 下一步动作；
- 到下一步动作的距离；
- 剩余距离；
- 总距离；
- 全路径 polyline。

手机端还做了一个很实际的优化：相同 payload 不重复发送。这减少了蓝牙带宽浪费，也降低了 HUD 端不必要的重复解析。

### 5.6 为什么手机端值得重点介绍

导航能力真正复杂的地方，从来不在“把箭头画出来”，而在：

- 如何稳定拿到高质量定位；
- 如何做路线缓存与投影；
- 如何判断偏航；
- 如何在网络与蓝牙状态波动时仍维持一个稳定、低抖动的导航体验。

`aura_nav` 在这些点上已经形成了较完整的工程实现，所以它不仅是一个配套 App，更是整个导航子系统的上游控制中心。

## 6. 后置摄像头功能

后置摄像头模块的目标很明确：给骑行/通勤场景提供实时后视信息，减少用户回头成本，提高态势感知能力。

从当前检查到的 `ar_app1_hud.cpp` 来看，默认参数是：

- 设备：`/dev/video21`
- 分辨率：`640 x 480`
- 默认采集帧率：`60 fps`
- 显示宽度：720 像素等效布局

代码本身已经体现出这是一个面向高帧率、低缓冲的设计：

- 优先使用 V4L2 打开摄像头；
- 设置 `CAP_PROP_BUFFERSIZE = 1`；
- 以独立线程持续采集；
- 在骑行模式中与导航、声音雷达一起组合显示。

你提到项目侧后置摄像头能达到 90 fps，这说明实际部署时硬件链路和驱动配置可能已经进入更高刷新模式。就我检查到的这份 `ar_app1_hud.cpp` 而言，当前默认值是 60 fps；如果板端实测已达 90 fps，那么说明真实运行配置比当前源码默认值更激进，这也是非常合理的工程演化方向。

广角后视对于 AR 骑行系统特别重要，因为它把“后方大范围感知”压缩成了一个可随时扫视的小窗口，和声音雷达一起，可以形成视觉 + 听觉的双模态安全辅助。

## 7. 声音转文字与文字转语音

### 7.1 使用 sherpa-onnx 生态进行本地化部署

语音相关能力建立在 sherpa-onnx 生态之上。你给出的 STT 模型来源页是：

- RK3588 流式双语语音识别模型页：
  <https://k2-fsa.github.io/sherpa/onnx/rknn/models.html#sherpa-onnx-rk3588-streaming-zipformer-small-bilingual-zh-en-2023-02-16>

在工程里，这套能力拆成了两部分：

- 语音转文字：`sherpa-onnx-alsa` + RK3588 Zipformer bilingual zh-en streaming small 模型；
- 文字转语音：`sherpa-onnx-offline-tts` / `sign_translate_tts_service.py` + Matcha zh-baker 声学模型 + Vocos vocoder。

所以更准确地说：STT 和 TTS 都来自 sherpa-onnx 这一开源项目生态，但使用的是不同的子模型链路。

### 7.2 语音转文字（STT）

STT 侧使用的模型文件包括：

- `encoder.rknn`
- `decoder.rknn`
- `joiner.rknn`
- `tokens.txt`

这类 Zipformer 架构模型的优势在于：

- 流式识别能力强，适合边说边出字；
- 双语中英场景适配更灵活；
- 通过 RKNN 部署到 RK3588 NPU，时延、功耗和离线能力都更适合 AR 设备；
- 不依赖云端，可降低隐私风险与弱网依赖。

在你的系统中，STT 不是单纯地把日志打印到终端，而是被整合到 `SPEECH TO TEXT` 和 `HEARING ASSIST` 模式里，为听障交流和骑行场景提供实时字幕能力。

### 7.3 文字转语音（TTS）

TTS 侧使用：

- `model-steps-3.onnx`
- `vocos-22khz-univ.onnx`
- `lexicon.txt`
- `tokens.txt`
- `phone.fst / date.fst / number.fst`

它在你的项目里有两条运行路径：

1. 通过 `sign_translate_tts_service.py` 驻留式常驻服务完成请求-响应式合成；
2. 若常驻服务不可用，则回退到 `sherpa-onnx-offline-tts` 命令行方式直接合成。

这套设计很成熟，体现了几个很有价值的工程优化：

- **常驻服务预热**：避免第一次调用时模型冷启动太慢；
- **缓存机制**：同一句话生成后的 wav 直接命中缓存；
- **重复播报去重**：短时间内不重复读相同文本；
- **多设备播放尝试**：自动检测 ALSA 播放设备，降低部署时音频设备不一致带来的问题。

### 7.4 接入 AR 项目后的改进点

相比把 sherpa-onnx 当一个独立 demo 使用，你在 AR 项目里做了更贴近产品化的适配：

- STT 和 TTS 都被纳入状态机统一管理；
- TTS 既服务语音播报，也服务手语识别结果输出；
- 听障交流模式下，手语识别文本和语音识别文本可以同时进入统一 UI；
- 通过缓存、常驻服务和线程化监督机制，尽量压缩了模型调用时延；
- 全链路保持离线运行，适合移动 AR 设备。

## 8. 声音雷达功能

### 8.1 8 路麦克风 + 声音分类 + 声源定位

声音雷达是这个项目里非常有特色的一部分。它不是简单的“拾音 + 阈值触发”，而是把：

- 8 路麦克风阵列采集；
- 声音事件分类；
- 声源方向估计；
- 时间平滑与异常抑制；
- HUD 方向可视化

完整地串成了一条链路。

### 8.2 声音分类模型

`sound_radar_service.py` 中使用的分类模型是：

- `yamnet_rk3588_int8.rknn`
- `yamnet_class_map.csv`

它基于 RKNNLite 在 RK3588 上运行。输入端会：

- 从 8 路麦克风中取分类专用通道；
- 重采样到 16 kHz；
- 构建 log-mel spectrogram；
- 生成 `(1, 96, 64, 1)` 形状的输入；
- 调用 RKNN 做推理。

在类别策略上，项目没有试图识别“所有声音”，而是聚焦真正对骑行/出行安全有价值的目标：

- 车喇叭相关（horn / honk / truck horn / toot 等）
- horn-like 警示声（buzzer / alarm / siren / bell / ringtone 等）
- 狗叫相关（bark / howl / growl / whimper 等）

对应阈值在代码里分开设置，例如：

- horn：0.20
- horn-like：0.12
- dog：0.30

这说明你在工程上做了“任务定制化裁剪”，而不是照搬通用音频分类器输出。

### 8.3 声音定位算法：不是黑盒，而是清晰的信号处理链

声音定位部分，代码中能读出一条相当完整的 DOA（Direction of Arrival）估计流程。

核心思路包括：

- 采样率 48 kHz，8 通道采集；
- 使用外圈 6 个通道做方向估计；
- 对信号做 300~4000 Hz 带通；
- 使用 GCC-PHAT 估计通道对时延曲线；
- 使用 SRP-PHAT 在 `-180° ~ 180°`、步长 `2°` 的角度空间内打分；
- 结合多通道几何布局计算最优方向；
- 通过直达声/回声能量比过滤反射干扰；
- 通过时间历史和置信度机制抑制抖动与离群值。

这说明你的声音雷达不是“神经网络一把梭”，而是把经典阵列信号处理与声音分类模型组合起来，形成了“先判别是什么，再估计从哪里来”的双阶段系统。

### 8.4 抗抖动与异常方向过滤

声音定位真正难的是环境反射、瞬时噪声和方向跳变。你的代码在这方面做了不少处理：

- 最近时间窗优先；
- 直达声/回声比过滤；
- 稳定 cluster 才接受；
- 冷启动确认机制；
- 大角度突变时的强跳变确认机制；
- 历史方向平滑与加权更新；
- 输出节流（0.25 秒）。

这几层机制叠加后，最终方向输出不会随着每一帧的瞬时最大值乱跳，而更接近一个“对用户有意义”的稳定方向提示。

### 8.5 HUD 呈现方式

HUD 侧接收到声音雷达事件后，会进一步做 UI 级平滑：

- 每个目标都有 `target_angle_deg` 和 `display_angle_deg`；
- 使用弹簧-阻尼模型推进显示角度；
- 限制角速度上限；
- 随时间衰减显示；
- 根据类别显示不同样式与颜色。

这使声音雷达不只是后台日志，而是一个真正适合 AR 头显展示的空间提示系统。对骑行和步行安全来说，这种“看不见的声音被转成可视方向”非常有价值。

## 9. 项目价值总结

这个项目最有意思的地方，不是某一个单独模型多先进，而是它把多种能力真正整合成了一套 AR 系统：

- 光学侧解决了近眼显示最难的畸变与双目汇聚问题；
- 渲染侧深入到底层硬件，追求低延迟和高效率；
- 交互侧把手语、语音、导航、声音、后视都接入同一显示平台；
- 模型侧充分利用 RK3588 的本地推理能力；
- 工程侧已经明显超出了“功能 demo”的范畴，开始具备产品原型系统的结构特征。

如果把它理解成一句话：这不是“AR 里塞几个功能”，而是在做一套围绕真实使用场景构建的、多模态、低延迟、可离线运行的 AR 交互平台。
